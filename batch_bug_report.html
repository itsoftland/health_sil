The Batch also shows 4 Sales Invoices in its Connections panel, matching the 3 deductions above (one may have been a partial or cancelled).

Observed vs Expected

What Was ObservedWhat Should Happen
Pharmacy Billing — when LITRAZ 5 is selected as item, the batch field is filled silently with MT-022575B. No popup appears. A popup (batch selection dialog) should appear showing that MT-022575B has only 1 unit remaining, highlighted in red as a low-stock warning.
No visual warning is shown at the point of batch selection. The user should see available qty before entering a billing quantity, so they know the constraint upfront.
Warning only appears after the user types a qty greater than 1 — at that point a "Insufficient Stock" message shows and qty resets to 0. The warning should be proactive — shown at batch-selection time — so the user is not left confused by their qty being reset to 0 unexpectedly.
Key Clarification
The stock data itself is correct and in sync. The SBB chain query returns qty_available = 1 accurately. The bug is entirely in the client-side JavaScript that decides when to show the batch selection dialog — not in the server-side stock calculation.

2. Where the Popup Data Comes From

Server-Side: get_batches_for_item()

File: health_sil/services/batch_selection_api.py, lines 6–55

This API is called by the client script when an item is selected. It queries the Serial and Batch Bundle (SBB) chain — the ERPNext v15 authoritative source for batch stock:

SELECT COALESCE(SUM(sbe.qty), 0) AS net_qty
FROM `tabSerial and Batch Entry` sbe
INNER JOIN `tabSerial and Batch Bundle` sbb ON sbe.parent = sbb.name
INNER JOIN `tabStock Ledger Entry` sle ON sle.serial_and_batch_bundle = sbb.name
WHERE sbe.batch_no  = %s
  AND sle.item_code = %s
  AND sle.is_cancelled = 0
  AND sbb.docstatus  = 1
-- sbe.qty: positive = inward (purchase), negative = outward (sale)
-- SUM gives correct net available qty

For batch MT-022575B:

  +15  (Purchase Invoice — opening stock)
   -5  (Sale, 29-05-2026)
   -2  (Sale, 29-05-2026)
   -7  (Sale, 10-06-2026)
  ────
   +1   ← net_qty returned to client

Since qty_available = 1 > 0, the batch IS included in the result. Only batches with qty_available > 0 are returned (line 52: if b.qty_available > 0: result.append(b)).

For LITRAZ 5, MT-022575B is the only batch with remaining stock — so the API returns a list with exactly one item.

Client-Side: show_batch_selection_dialog()

File: Client Script — "Pharmacy Billing Conditions" (Enabled: 1), lines 594–771

This function receives the batch list from the server and decides what to show. It has two paths:

ConditionWhat HappensLow-Stock Warning?
batches.length === 0 Shows "No stock available for this item" msgprint Yes (item level)
batches.length === 1 Auto-selects batch silently. Dialog skipped. No warning shown. NO
batches.length > 1 Shows full batch selection dialog with available qty per batch. Qty ≤ 10 shown in red. Yes (batch level, visual)

MT-022575B falls into the batches.length === 1 path — the only path with no warning at all.

qty Trigger: validate_batch_qty()

File: Client Script — "Pharmacy Billing Conditions", lines 507–527
File: batch_selection_api.py, lines 59–100

After the batch is auto-selected, when the user types a quantity, the qty trigger calls validate_batch_qty(). This fires a second independent server call and will correctly block qty > 1. But this is after the fact — the user had no way to know the constraint before typing.

What This Means for MT-022575B

  • Only one batch exists with stock for LITRAZ 5 → batches.length === 1
  • The if block fires immediately and fills the batch field silently
  • The variable b.qty_available = 1 is never rendered to the screen
  • The avail-low / avail-ok red/green colouring logic at line 624 is never reached
  • The batch selection dialog (with its qty-exceeds-available guard at lines 689–693) is never opened

What Multi-Batch Items Get That Single-Batch Items Don't

If LITRAZ 5 had two batches, the dialog at lines 622–769 would display with this per-row logic:

// 'avail-low' → renders qty in RED in the dialog table
// MT-022575B with qty=1 WOULD be red — but this line is never reached

Impact
A pharmacy staff member selecting LITRAZ 5 today will see the batch auto-filled with no indication that only 1 unit remains. They will not discover the constraint until after they type a quantity — at which point their entry is silently reset to 0, causing confusion. There is no way to know the available quantity without opening the Batch document separately.

4. Bug 2 — Low-Stock Warning Is Reactive, Not Proactive

BUG-2HIGH

The qty Trigger (The Only Warning That Does Fire)

Client Script: "Pharmacy Billing Conditions", lines 507–527

qty: function (frm, cdt, cdn) {
    const row = locals[cdt][cdn];
    if (!row.qty || row.qty <= 0) return;

    if (row.batch) {
        frappe.call({
            method: 'health_sil.services.batch_selection_api.validate_batch_qty',
            args: { batch_no: row.batch, qty: row.qty },
            callback: function (r) {
                if (r.message && !r.message.ok) {
                    frappe.msgprint({
                        title: __('Insufficient Stock'),
                        message: __(r.message.message),   // "Only 1 Nos available for this batch."
                        indicator: 'red'
                    });
                    frappe.model.set_value(cdt, cdn, 'qty', 0);  // ← resets to 0 silently
                    return;
                }
                apply_row_calculations(frm, cdt, cdn);
            }
        });
    }
}

Why This Alone Is Insufficient

AspectCurrent BehaviourProblem
Timing Warning fires only after user finishes typing qty and tabs away User has already committed intent; the silent reset to 0 is confusing
Information User must type a number to discover the available qty User has no way to know the constraint at batch selection time
UX Qty field resets to 0 after the message is dismissed User may re-type the same number, triggering the same loop
Discoverability Available qty (1) never shown in the form There is no field on the Pharmacy Billing form that shows batch qty to the user
Why the Warning Exists But Doesn't Help Enough
The validate_batch_qty server call is correctly coded and the message text is accurate ("Only 1 Nos available for this batch"). The problem is when it runs — after the user has already decided on a quantity — rather than before the user enters a quantity, which is when the information is most useful.

5. Bug 3 — Race Condition Between Two Server Calls

BUG-3MEDIUM

The Two Independent Server Calls

The client script makes two separate, independent server-side queries for batch stock at different points in the workflow. They do not share state.

CallFunctionTimingReturns
Call 1 get_batches_for_item() When user selects item_code List of batches with qty_available per batch
Call 2 validate_batch_qty() When user changes the qty field { ok, available, message }

The Race Condition

There can be a time gap of several minutes between Call 1 (item selected) and Call 2 (qty entered). In a busy pharmacy, another billing transaction for LITRAZ 5 MT-022575B can complete in that window:

User A opens Pharmacy Billing, selects LITRAZ 5 ↓ Call 1: get_batches_for_item() → { batch: 'MT-022575B', qty_available: 1 } ↓ (User A is still thinking / filling other fields) ↓ User B submits a Sales Invoice for MT-022575B, qty=1 → Stock Ledger Entry created → SBB net qty is now 0 ↓ User A enters qty = 1 ↓ Call 2: validate_batch_qty('MT-022575B', 1) ↓ Server returns: { ok: false, available: 0, message: "Insufficient stock. Only 0 Nos available" } ← But Call 1 had shown qty_available=1 to User A ! ↓ User A sees qty reset to 0 — confused because the batch showed 1 available moments ago

Why This Matters in the Current Scenario

With only 1 unit remaining, any concurrent billing will push stock to 0. The gap between Call 1 and Call 2 is the window of vulnerability. This is a low-frequency but high-confusion bug — the user cannot understand why a batch that "had stock" is now blocked.

Note — This Does Not Cause Overselling
The final safety net is in generate_invoice_api.py — the Sales Invoice submission with update_stock=1. ERPNext will prevent negative stock at the SLE level. The race condition causes a confusing UX error, not a data integrity error.

6. Bug 4 — Disabled Duplicate Script

BUG-4LOW

The "Pharmacy Batch Selection Dialog" Client Script

There is a second client script attached to the Pharmacy Billing DocType:

Script NameDocTypeStatus
Pharmacy Billing ConditionsPharmacy BillingEnabled: 1 (Active)
Pharmacy Batch Selection DialogPharmacy BillingEnabled: 0 (Disabled)

The disabled script was the original batch selection implementation. It was disabled when the merged "Pharmacy Billing Conditions" script (v6/v7) was created. The disabled script also listened on the item_code trigger in the Pharmaceuticals child table and called get_batches_for_item().

Why This Matters

  • The disabled script had the same single-batch silent auto-select bug — so this is not a regression introduced in v6/v7. The warning was never built for the single-batch path in either version.
  • The disabled script is still present in the fixtures (client_script.json). If it is accidentally re-enabled (e.g. during a fixture import or a rollback), both scripts will run simultaneously on the same item_code trigger, causing the batch dialog to open twice for multi-batch items.
  • The disabled script also had a separate batch: field trigger that fetched MRP from the Batch document when batch was manually changed. This trigger was merged into the active script's logic but exists as dead code in the disabled version.
Risk of Accidental Re-Enable
If bench migrate or import-doc re-imports the fixtures and the enabled flag is set to 1 in the JSON by mistake, the disabled script becomes active. Both scripts would then fire on item_code, opening two dialogs for multi-batch items and auto-selecting the batch twice for single-batch items. The _fetching_rows Set guard in the active script would partially prevent double-firing, but only for the same row — cross-script timing is not guarded.

The user experiences a confusing workflow: batch is selected automatically (no choice shown), they enter a qty, the qty is silently reset to 0, and then they see a message telling them there is only 1 unit. The correct quantity (1) was always known to the system from Step 6, but was never surfaced to the user.
