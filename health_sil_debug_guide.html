1. Complete Pharmacy Billing Workflow

The billing workflow is triggered entirely from the client side (JavaScript). There is no on_submit hook on Pharmacy Billing in hooks.py — the JS form calls the whitelisted Python API directly.

End-to-End Call Chain

2. Why Sales Invoice Is Not Being Created

Root Cause Summary
The failure is silent by design. The handle_errors() function in generate_invoice_api.py:30–31 catches ALL exceptions, logs the full traceback to frappe.log_error, then throws a generic "Process failed" message to the user. The Sales Invoice was never created — but the error detail is in the Error Log.

The Silent Failure Pattern (Code Reference)

# generate_invoice_api.py — lines 30-33
except Exception as e:
    handle_errors(e, original_user)   # <-- logs full traceback, throws generic msg
finally:
    frappe.set_user(original_user)    # <-- ALWAYS resets user session

The handle_errors() function at line 237–249 logs: the user email, their roles at time of failure, the error message, and the full Python traceback. All of this is in the Frappe Error Log.

How to Read the Error Log

# Option 1 — Live tail from bench directory:
cd ~/frappe-bench
tail -f sites/your-site/logs/frappe.log | grep -A 20 "Pharmacy Billing Error"

# Option 2 — Frappe Desk UI:
# Settings > Error Log > filter Title = "Pharmacy Billing Error"
# OR Title = "Payment Processing Failed"

# Option 3 — bench console query:
bench --site your-site console
>>> import frappe
>>> logs = frappe.get_all("Error Log",
...     filters={"title": "Pharmacy Billing Error"},
...     fields=["name", "creation", "error"],
...     order_by="creation desc", limit=5)
>>> print(logs[0].error)   # Full traceback of the last failure

Most Likely Root Cause After Permission Change

When a user with a restricted role submits a Sales Invoice with update_stock=1 and a batch_no, ERPNext internally calls make_serial_and_batch_bundle(). That internal function calls doc.save() and doc.submit() without setting ignore_permissions=True. This is documented in the codebase at setup_permissions.py:17–19:

Critical Permission — Serial and Batch Bundle
ERPNext's make_serial_and_batch_bundle does doc.save() and doc.submit() WITHOUT setting ignore_permissions. The calling user therefore needs read, write, create, and submit permissions on Serial and Batch Bundle. If this was revoked, every pharmacy invoice silently fails at the stock deduction step. 
# setup_permissions.py — line 36 (the critical permission entry):
{"doctype": "Serial and Batch Bundle", "permlevel": 0,
 "read": 1, "write": 1, "create": 1, "submit": 1},

Check if Invoice Was Created but Payment Failed

process_payment() wraps payment errors separately (line 161–168). The invoice may exist even when the user saw an error. Always check:

...     filters={"patient_name": "PAT-0001"},   # replace with actual

4. Permission Analysis

How the App Bypasses Permissions (Design Intent)

The app uses a two-step pattern for billing APIs to allow restricted roles to create invoices without granting them broad access:

  1. API runs as original_user for initial validation (Customer lookup)
  2. API switches to Administrator via frappe.set_user("Administrator")
  3. Invoice/Payment Entry created with ignore_permissions=True
  4. Ownership patched back to original user via frappe.db.set_value

This is safe and intentional — it preserves the audit trail while bypassing ERPNext's often over-restrictive permission model for nested document operations.

What Permissions Are Required and Why

Serial and Batch Bundleread, write, create, submitERPNext calls doc.save() + doc.submit() with NO ignore_permissions
Customerreadget_validated_customer() runs as original user BEFORE Administrator switch

AccountreadInvoice validation calls _get_party_details() internally
WarehousereadWarehouse lookup in batch queries
Mode of Paymentreadvalidate_mode_of_payment() checks MoP exists

The setup_permissions.py function is idempotent — running it multiple times is safe. It checks for existing rows before creating, and updates existing rows if flags were missing.

The setup function creates Custom DocPerm rows (not core DocPerm). Frappe preserves custom permissions across framework updates, so your role permissions will not be wiped by bench update. The function is also called from the after_migrate hook, so it re-runs automatically after every migration.

5. Batch & Stock Issue Analysis

Two Parallel Mechanisms — The Core Problem

The codebase contains two separate mechanisms for tracking batch stock. When both run on the same transaction, stock quantities diverge.

MechanismFile / LineHow It WorksAuthoritative?
A — Sales Invoice SLE generate_invoice_api.py:69 Sets update_stock=1. ERPNext creates SBB → SBE → SLE on submit. Deducts stock via proper accounting entries. Yes — this is correct
B — Direct batch_qty batch_selection_api.py:104 Directly sets Batch.batch_qty via frappe.db.set_value. No SLE, no SBB created. Commits immediately. No — deprecated shortcut
Double Deduction Risk
If a client script calls deduct_batch_stock() AND the Sales Invoice is created with update_stock=1 for the same batch, the stock is deducted twice: once in Batch.batch_qty and once via SLE. The batch_qty field will show less stock than actually exists.

How Batch Availability Is Queried (ERPNext v15)

In ERPNext v15, SLE.batch_no is always NULL. Batch quantities live in the Serial and Batch Entry table. The correct query pattern (used in batch_selection_api.py:35–44):

SELECT COALESCE(SUM(sbe.qty), 0) AS net_qty
FROM `tabSerial and Batch Entry` sbe
INNER JOIN `tabSerial and Batch Bundle` sbb ON sbe.parent = sbb.name
INNER JOIN `tabStock Ledger Entry` sle ON sle.serial_and_batch_bundle = sbb.name
WHERE sbe.batch_no  = %s
  AND sle.item_code = %s
  AND sle.is_cancelled = 0
  AND sbb.docstatus  = 1
-- sbe.qty: positive = inward, negative = outward
-- SUM gives net available quantity

For batches imported directly (no purchase invoice, no SBB), the fallback is Batch.batch_qty.

Diagnostic SQL — Detect Divergence

Run this in bench console or directly in MySQL to find batches where batch_qty disagrees with actual SBB stock:

bench --site your-site mariadb
-- or in bench console: frappe.db.sql("""...""", as_dict=True)

SELECT
    b.name          AS batch_no,
    b.item,
    b.batch_qty     AS stored_batch_qty,
    COALESCE(SUM(sbe.qty), 0) AS sbb_net_qty,
    (b.batch_qty - COALESCE(SUM(sbe.qty), 0)) AS divergence
FROM `tabBatch` b
LEFT JOIN `tabSerial and Batch Entry` sbe ON sbe.batch_no = b.name
LEFT JOIN `tabSerial and Batch Bundle` sbb
    ON sbe.parent = sbb.name AND sbb.docstatus = 1
LEFT JOIN `tabStock Ledger Entry` sle
    ON sle.serial_and_batch_bundle = sbb.name AND sle.is_cancelled = 0
WHERE b.item = 'YOUR-ITEM-CODE'   -- replace
GROUP BY b.name, b.batch_qty
HAVING ABS(divergence) > 0;
-- divergence != 0 means deduct_batch_stock() ran without a matching SLE

Trace a Specific Batch's Complete History

bench --site your-site console
>>> import frappe
>>> batch_no = "BATCH-0001"   # replace with actual
>>> sles = frappe.db.sql("""
...     SELECT sle.posting_date, sle.voucher_type, sle.voucher_no,
...            sle.actual_qty, sle.qty_after_transaction, sle.warehouse
...     FROM `tabStock Ledger Entry` sle
...     WHERE sle.item_code = (
...         SELECT item FROM `tabBatch` WHERE name = %s
...     )
...     AND sle.is_cancelled = 0
...     ORDER BY sle.posting_date, sle.creation
... """, batch_no, as_dict=True)
>>> for s in sles: print(s)

Fix Direction

  • If deduct_batch_stock() is called from a client script alongside invoice creation: remove the client-side call. The SLE from update_stock=1 is the single source of truth.
  • If batches were imported and have no SBB history, the batch_qty fallback in batch_selection_api.py:50 is correct and intentional.
  • Do NOT manually update batch_qty to fix divergence — create a corrective Stock Reconciliation entry instead, which will generate the proper SLE and keep the audit trail intact.

6. Code Fix Risk Analysis

Based on the patterns in the existing codebase, here are the four highest-probability risk areas to evaluate against any fix. Share your code snippet for a precise review against these points.

Risk Area 1 — Administrator Switch Scope

generate_invoice_api.py:15–20

original_user = frappe.session.user
try:
    customer = get_validated_customer(patient_name)   # <-- RUNS AS original_user
    frappe.set_user("Administrator")                  # <-- switch happens HERE
Risk
Any code your fix adds before frappe.set_user("Administrator") runs as the restricted original user. If that code reads a protected doctype, it fails silently in the except block. If your fix moves the Administrator switch later, more code runs as the restricted user than intended.

Risk Area 2 — The finally Block Guarantees User Reset

generate_invoice_api.py:33

finally:
    frappe.set_user(original_user)   # ALWAYS runs, even on exception
Safe Pattern
A return inside the try block still triggers finally. However, if your fix introduces a second frappe.set_user() call inside the try block without a matching finally, a mid-function exception will leave the session as that user until the request ends.

Risk Area 3 — Double Commit in Batch Operations

batch_selection_api.py:124

frappe.db.set_value("Batch", batch_no, "batch_qty", new_qty)
frappe.db.commit()   # <-- commits immediately, outside any outer transaction
Risk
If your fix calls deduct_batch_stock() from inside create_and_submit_invoice(), the frappe.db.commit() at line 124 fires mid-transaction. If invoice.submit() later fails, the database rolls back the invoice but batch_qty has already been committed. You now have a stock count that is lower than the actual SLE-tracked quantity.

Risk Area 4 — ignore_permissions Does Not Cover SBB

generate_invoice_api.py:77 and setup_permissions.py:34–36

# invoice.insert(ignore_permissions=True) covers the Sales Invoice document.
# But ERPNext's make_serial_and_batch_bundle() inside invoice.submit()
# calls its own doc.save() and doc.submit() — WITHOUT ignore_permissions.
# Only the user's role permissions on Serial and Batch Bundle matter there.
Risk
If your fix removes the frappe.set_user("Administrator") call and instead relies only on ignore_permissions=True on the invoice, the invoice insert will succeed but the SBB creation triggered by submit will fail for any user without the explicit Serial and Batch Bundle submit permission.

Safe Production Debugging Workflow

  1. Read Error Log first — never guess; the traceback is always in Frappe Error Log
  2. Reproduce on dev site — use bench --site dev-site with a copy of production data
  3. Export fixtures before any change — bench export-fixtures --app health_sil + git commit
  4. Run permission setup idempotently — setup_custom_role_permissions() is safe to call anytime
  5. Clear cache after permission changes — Frappe caches permissions aggressively; a stale cache can make valid permissions appear broken
  6. Never modify core Frappe/ERPNext files — use Custom DocPerm for permissions, Custom Script for JS, server scripts for Python overrides

Scheduler / Background Job Debugging

# Check if scheduler is running:

# View queued/failed jobs:
bench --site your-site jobs
